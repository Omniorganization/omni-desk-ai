# Industrial Runtime Optimization — 2026-07-17

This change closes the highest-value source-level optimization gaps identified after the
dependency-upgrade merge train.

## Implemented

- Bounded shared PostgreSQL connection pool for core state and transactional outbox.
- Cached `PostgresRuntimeStateStores` and outbox repositories per runtime factory.
- SQL-filtered, `FOR UPDATE SKIP LOCKED LIMIT 1` queue/outbound claims.
- Indexed approval lookup and ready-work selection.
- Five-second shallow readiness cache executed outside the ASGI event loop.
- Separate deep authenticated readiness path.
- Deep readiness validates all Transactional Outbox columns on every probe, bypassing
  the process-local schema cache so post-startup schema loss or drift fails closed.
- PostgreSQL pool pressure metrics exposed through runtime/readiness status.
- Canonical channel adapter instances reused by user-facing aliases.
- Repository factory added to deterministic runtime resource cleanup.

## Risk notes

- An undersized PostgreSQL pool can increase acquisition latency or exhaust the configured
  wait timeout during bursts. Operators must monitor created, in-use, idle, and waiter
  metrics and size the pool against replica count and PostgreSQL connection limits.
- The five-second shallow readiness cache can delay visibility of a new database failure by
  up to five seconds. The authenticated deep readiness endpoint remains uncached and is the
  authoritative diagnostic path.
- SQL-level claims change contention behavior from process-side filtering to row locking.
  Rollout must verify claim throughput, stale-lease recovery, duplicate suppression, and
  dead-letter behavior under concurrent workers.
- Shared repository lifetime means shutdown ordering matters. The runtime closes state
  stores before the shared pool and regression tests cover deterministic cleanup.
- Mobile model/security extraction changes source boundaries without changing persisted
  session keys or API contracts; native build and simulator gates remain mandatory.

## Rollback procedure

1. Stop or drain new application instances before reverting so no worker is interrupted
   while holding an outbox or queue lease.
2. Revert the squash commit for PR #93 and deploy the previous `main` revision. This change
   contains no destructive database migration, so existing runtime and outbox rows remain
   compatible with the prior implementation.
3. Remove or restore the previous values of `OMNIDESK_CORE_POSTGRES_POOL_SIZE` and
   `OMNIDESK_CORE_POSTGRES_POOL_TIMEOUT_SECONDS`, then restart all replicas to discard the
   shared in-process pools.
4. Verify `/ready` and authenticated `/admin/ready`, confirm queue and outbox dispatchers are
   processing, and inspect retry/dead-letter counts for work interrupted during the drain.
5. Run the repository smoke, PostgreSQL atomicity, security, and Tri-App gates against the
   rollback revision before reopening traffic.
6. If rollback was triggered by database symptoms, capture a backup and run the existing
   backup/restore validation before resuming production writes.
